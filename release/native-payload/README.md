# Native Payload Release Boundary

This directory owns the public, non-secret trust policy for the macOS Native payload release. It
does not contain release credentials.

## Current Status

The production release path is fail-closed and intentionally not provisioned. Do not add placeholder
trust values to make the workflow pass. A release owner must approve and commit all three public
policy files before the protected workflow can run:

- `allowed_signers`: one Ed25519 OpenSSH allowed-signers entry whose principal matches the manifest
  verifier and the portable public shell-bootstrap contract;
- `apple-team-id`: the exact 10-character Apple Developer Program team identifier authorized for
  the public release.
- `mongodb-redistribution-approved`: the reviewed public distribution approval required by the
  license matrix before a candidate containing MongoDB may be uploaded.

Their corresponding private keys, certificate password, and Apple notarization credentials belong
only in the protected `native-payload-release` GitHub environment. They must never be committed,
included in a candidate artifact, or accepted from public-bootstrap environment variables.

## Candidate Producer Contract

The protected `.github/workflows/native-payload-release.yml` workflow consumes a successful
same-repository, same-commit Actions run. That producer must upload exactly one artifact per
architecture:

- `native-payload-root-arm64`
- `native-payload-root-x86_64`

Each artifact contains `payload/` and `bootstrap/ViventiumBootstrap.app`. The payload contains only
regular, non-hard-linked files in mode `0644` or `0755` and is complete enough to run the Native
core without Git, system Python/Node, package managers, or build tools. It also contains:

- a policy-selected minimal Node runtime (`bin/node` plus its exact license), minimal MongoDB
  runtime (`bin/mongod` plus publisher notices), and a materialized minimal Python runtime without
  pip, ensurepip, venv, Tk, tests, development files, or generated bytecode;
- identical deterministic `python-runtime-manifest.json` files beside the payload and Bootstrap
  copies, binding every path, mode, size and SHA-256 digest to the exact component policy;
- a non-served, hash-bound compiled-client compliance closure produced from the exact rendered
  Rollup modules before browser dependencies are pruned. Every contributing package-lock root and
  vendored adapter must retain package-owned license/notice bytes; a missing or ambiguous record is
  a release failure, not an inferred approval;
- `payload/release-metadata/apple-code-paths.txt`: safe relative paths ordered from the innermost
  code object to the enclosing bundle;
- `payload/release-metadata/apple-staple-paths.txt`: safe relative paths for notarized containers
  that support stapling.

`.github/workflows/native-payload-candidate.yml` is the exact dual-architecture producer. It verifies
public component pins and digests, builds and prunes LibreChat, compiles canonical runtime config,
builds the helper/bootstrap apps, assembles a relocatable root, and runs install, start, registration,
Connected Accounts start, health, and stop smoke checks. It receives no release credentials. The
producer also refuses to run unless `components.lock.json` and `components.json` both mark their
publication state as `merged`; review-head pins remain explicit and cannot become an artifact by
accident. OpenClaw remains `lab-only` and unapproved in the public component manifest. The
protected release workflow injects only public trust/release policy, signs nested code and both apps,
notarizes/staples supported bundles, packages the bootstrap archive, independently re-verifies it,
and allowlists the exact draft-release assets.

Immutable Native LibreChat does not bind the historical API port `3180`. Its backend and public
`3190` proxy communicate through the exact owner-checked Unix socket in the private Native runtime
directory, including first-admin registration and WebSocket upgrades. Candidate smoke requires no
listener on `3180`; source and Docker profiles keep their separate documented TCP contracts.
Immutable Native MongoDB is also socket-only: `mongod` binds the exact support-owned mode-`0600`
Unix socket, disables its automatic `/tmp` socket, and is rejected if its process group exposes any
TCP listener. Source and Docker profiles retain their separate configured MongoDB TCP contracts.

The producer is intentionally fail-closed until MongoDB redistribution approval is recorded. The
release workflow remains fail-closed until all three public policy files and protected authorities
exist. Source completeness is therefore not evidence of a signed, notarized, or published release.

Native publication is globally serialized. The protected workflow signature-verifies every retained
GitHub release that carries a Native bootstrap index, rejects any gap in the complete `1..N`
sequence, and accepts only sequence `1` or exactly the prior high-water sequence plus one. The tagged
`install.sh` must embed the approved signer key,
Apple team, and a minimum sequence equal to the candidate; empty or mismatched values stop before
signing. The outer index, signed app policy, and signed payload manifest must carry the same release
identity and sequence, while established installs keep an owner-only high-water mark. A pristine
machine still needs an independently authenticated current bootstrap/freshness boundary to rule out
replay of an older valid shell plus its older signed release; this repository does not invent that
external authority.

## Local QA Versus Production

`scripts/viventium/assemble_native_payload.py --mode local-qa` and
`scripts/viventium/build_native_payload.py --channel local-qa` create a relocatable local root and a
deterministic, deflated, explicitly unsigned local artifact. Reproducibility is within the pinned
build interpreter/toolchain, not a cross-toolchain promise. The verifier accepts the archive only
with its explicit local-QA override. These paths are useful for assembly, install, archive, staging,
activation, health, recovery, and rollback tests; they are never public-release evidence.

The `stable` channel requires an SSH manifest signing key. The protected release workflow additionally
requires Developer ID signing with hardened runtime and timestamping, Apple notarization with a
successful `--wait` result, stapling and Gatekeeper checks, dual-architecture installed-artifact
health checks, attestations, and a complete GitHub draft release. Publishing that draft remains a
separate release-owner decision after clean-Mac acceptance.

The immutable Native edition exposes the **Easy Install** lifecycle. Its public CLI supports
launch/start/stop/status/doctor, local one-time `password-reset-link`, complete owner-only `snapshot`,
same-profile transactional `restore`, and ownership-safe uninstall. Snapshot and restore ship the
same logical-bundle validator and Mongo adapter as the source continuity flow; restore stages on the
App Support filesystem, journals exact-root activation, automatically rolls back injected failure,
and never replaces the immutable release pointer or machine secrets. Restored credentials remain
excluded and require reconnect/password recovery; Recall requires rebuild. Custom Settings Install
remains available through the source installer. Native in-place upgrade and cross-profile migration
remain unavailable. Install Native updates by running a newly verified publisher-signed Bootstrap.

## External Administration Required

Before production use, a repository administrator must configure the `native-payload-release`
environment, restrict its deployment branch/tag policy, provision its secrets, and enable GitHub
immutable releases. A multi-member organization should add required reviewers and prevent
self-review. A sole-owner organization must leave required reviewers off to avoid an impossible
self-approval queue; the release remains an explicit owner-dispatched workflow from protected,
reviewed source and still fails closed on CI, sequence, provenance, signing, notarization, and
artifact checks. The workflow checks the authorities and immutable-release API, but repository
settings cannot be safely invented or enforced from workflow YAML alone.
