# Scheduled Heartbeat Keepalive Suppression QA - 2026-05-02

## Scope

Validate the scheduler visibility contract exposed by the heartbeat incident: any scheduled Telegram
run that produces `{NTA}` or empty output must remain silent instead of receiving scheduler-authored
status text.

## Sanitized Incident Shape

- Surface: scheduled Telegram delivery.
- Schedule type: passive scheduled check.
- Canonical model result: `{NTA}`.
- Observed user-visible output: a scheduler-authored "quick pulse" status message with next-check
  timing.
- Runtime classification before fix: `sent` / `heartbeat_keepalive`.

## Root Cause

The scheduler dispatch layer had a heartbeat keepalive override. After repeated suppressed `{NTA}`
heartbeat runs, it replaced silence with a hardcoded user-visible Telegram follow-up and recorded the
delivery as `heartbeat_keepalive`.

This violated the Scheduling Cortex contract: passive scheduled runs with no substantive update
should remain silent, and Telegram should deliver only canonical scheduler-generated content. The
contract is generic; it must not depend on a task being named heartbeat.

The load-bearing runtime fix is in dispatch visibility and channel fan-out, not only scheduler
persistence. The old dispatch path both synthesized the keepalive text and stamped the
`heartbeat_keepalive` reason; scheduler persistence also wrote quiet-streak metadata that made the
dispatch override possible.

## Fix Contract

- The scheduler no longer synthesizes heartbeat keepalive text.
- Legacy heartbeat metadata, if present in any local schedule row, is inert and does not change
  delivery behavior.
- Any scheduled task that returns `{NTA}` remains suppressed even if stale quiet-streak metadata is
  present.
- Explicit user-visible reminders and substantive scheduled outputs still deliver normally.
- Scheduler runtime metadata such as next run time must not be turned into user-facing Telegram text.
- Runtime behavior must not branch on schedule names, prompt text, or incident-specific labels.

## Residual

- Historical schedule rows can still contain a prior `heartbeat_keepalive` delivery ledger. The fix
  does not mutate existing user data in-place; stale quiet-streak metadata is pruned only on a future
  successful scheduler update and is no longer used for delivery behavior.
- The default system Python in this local environment did not have `pytest` installed, so the root
  release test was executed through `uv run --with pytest`.
- Local runtime is fixed and loaded, but this is not release-ready for future installs until the
  LibreChat sub-repo changes are committed and the parent component pin is bumped.
