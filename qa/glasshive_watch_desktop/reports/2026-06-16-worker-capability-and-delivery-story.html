GlassHive QA Story

Worker capabilities, live workspace, and callback delivery

2026-06-16

1. Flow

User or Host AI prompt
↓
LibreChat / GlassHive MCP / GlassHive UI
↓
GlassHive bootstrap gives files, context, native capability awareness, callback
↓
Selected worker CLI: Codex or Claude
↓
Worker chooses native skills/tools when relevant
↓
Watch / Steer, file actions, chat callback

2. RCA

AWatch desktop was not trustworthy. A completed file could take over the desktop frame, and noVNC was resetting because websockify inherited a mounted worker-home TMPDIR.

BLibreChat result visibility was unreliable. Callback delivery and message persistence worked, but the owning conversation message list / updatedAt was stale.

ImportantThe worker produced useful output. The broken layer was the delivery and live-surface substrate around the worker.

3. Capability Injection References

  • bootstrap.py lines 29-89: operating instructions, native capability inventory, completion/self-review contract.
  • bootstrap.py lines 168-188: writes AGENTS.md, CLAUDE.md, and CODEX.md into the worker workspace.
  • docker_sandbox.py lines 142-179 and 625-689: service temp path, noVNC health, self-heal.
  • watch.js lines 188-193 and 520-539: desktop stays desktop; file deliverables stay explicit actions.
  • glasshive.js lines 593-610: visible callback persistence touches conversation metadata.

4. Fixes

Fixed
Desktop surface fidelity
surface=desktop keeps the workstation desktop as the primary frame.
Fixed
noVNC health
New containers use service-local /tmp; runtime checks /core/rfb.js and repairs noVNC before declaring the view available.
Fixed
Callback visibility
Visible callbacks update the conversation so refresh/reopen surfaces the result.
